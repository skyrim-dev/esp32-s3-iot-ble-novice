# 待实现功能

本文档整理当前项目中**还没有实现**、但在前面分析中已经明确值得后续补充的功能。内容只基于当前代码状态编写。

---

## 1. 细化 OTA 失败原因映射

### 当前状态

在 `components/service/ota/hw_iot_ota.c` 中，OTA 成功时会上报：

- `result_code = 0`

失败时当前基本统一处理为：

- `result_code = 255`
- `description = "ota installation failed"`

### 后续建议

把 `esp_https_ota()` 返回的错误进一步映射成更具体的 OTA 状态码，例如：

- 下载超时
- 升级包校验失败
- 安装失败
- 内存不足
- 内部异常

### 价值

- 平台侧更容易定位失败原因
- 设备日志和云端状态能对应起来

---

## 2. 收紧 OTA 下载的证书校验

### 当前状态

在 `components/service/ota/hw_iot_ota.c` 中，OTA HTTP 配置里有：

```c
.skip_cert_common_name_check = true
```

注释已经说明这是测试环境用法。

### 后续建议

- 去掉该测试配置，或
- 做成 Kconfig 可配置项，并默认关闭

### 价值

- 提高 OTA 下载链路的安全性
- 更接近生产环境要求

---

## 3. 增加更明确的 OTA 状态枚举/状态机

### 当前状态

当前 OTA 流程已经有这些实际状态：

- 开始升级
- 安装完成，待重启
- 重启后确认成功
- 升级失败

但这些状态目前分散在：

- `hw_iot_ota.c`
- `hw_iot_mqtt_client.c`
- OTA 状态上报 description
- NVS 的 `reboot_pending` 标志

### 后续建议

增加一个统一的 OTA 状态枚举，例如：

- `IDLE`
- `DOWNLOADING`
- `INSTALLING`
- `REBOOT_PENDING`
- `CONFIRMED`
- `FAILED`

### 价值

- OTA 流程更清楚
- 后续扩展错误处理、重试、回滚时更容易维护

---

## 4. 接入 ESP-IDF 官方 rollback / app valid 流程

### 当前状态

当前项目已经实现了：

- 双 OTA 分区
- OTA 成功后重启
- 重启后二阶段确认

但代码里还没有看到这些官方回滚接口：

- `esp_ota_mark_app_valid_cancel_rollback()`
- `esp_ota_mark_app_invalid_rollback_and_reboot()`

### 后续建议

后续可以考虑：

- 新固件第一次启动后先做自检
- 自检通过则标记 app valid
- 自检失败则回滚到旧固件

### 价值

- 避免“新固件能启动但不稳定”时设备卡死在坏版本
- OTA 可靠性更高

---

## 5. 提高 OTA 状态上报的可靠性

### 当前状态

在 `components/protocol/hw_iot_mqtt/hw_iot_mqtt_publish.c` 中，当前 MQTT 发布是：

- `QoS = 0`

也就是说，当前更偏向“尽力发送”，不是强确认型上报。

### 后续建议

可以评估后续是否要：

- 对关键 OTA 状态使用更高可靠性的发布策略
- 或增加更清晰的发布确认逻辑

### 价值

- 重要 OTA 状态更不容易丢失
- 平台显示结果更稳定

---

## 6. 增加更细粒度的 OTA 进度上报（可选）

### 当前状态

当前项目已经能上报 OTA 状态，但更偏向关键阶段结果：

- 版本查询
- 安装完成待重启
- 重启后确认成功

尚未实现更细的下载/写入进度百分比。

### 后续建议

如果后面继续深入 OTA，可以考虑增加：

- 下载开始
- 下载中
- 安装中
- 安装完成
- 重启确认完成

### 价值

- 更适合调试 OTA
- 平台侧可观测性更好
