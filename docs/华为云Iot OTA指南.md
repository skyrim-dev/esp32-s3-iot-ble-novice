# 华为云 IoT OTA 指南

## 文档概述

本文基于华为云官方文档整理，面向本项目的设备端 OTA 实现说明。

官方参考文档：

- [MQTT设备OTA升级](https://support.huaweicloud.com/usermanual-iothub/iot_01_0047.html)

本文重点说明以下内容：

- 华为云 MQTT OTA 的标准流程
- 设备端在每个阶段需要做什么
- 平台为什么可能在“仅上报版本”后就显示任务成功
- 在 ESP32 项目中实现完整 OTA 还需要补哪些能力

## OTA 标准流程

根据华为云官方文档，MQTT 设备 OTA 的完整流程如下：

1. 用户在华为云平台上传软件包/固件包。
2. 用户在平台创建升级任务。
3. 平台感知设备在线后，触发升级协商流程；如果设备离线，则等待设备上线。
4. 平台向设备下发“查询当前软件版本”的指令。
5. 设备返回当前版本号，平台根据目标版本判断是否需要升级。
6. 如果需要升级，平台向设备下发升级包下载地址、Token 以及相关信息。
7. 设备通过 HTTPS 下载升级包。
8. 设备执行升级，升级完成后向平台反馈结果。
9. 平台将升级结果同步到控制台或应用侧。

## 关键判断点

### 1. 版本是否一致

这是最关键的判断点。

- 如果设备上报的当前版本 **与升级任务目标版本相同**，平台会直接判定本次升级任务成功。
- 如果设备上报的当前版本 **与目标版本不同**，平台才会继续下发下载地址和升级信息。

这也是你在调试时最容易误解的一点：

> 平台任务显示“成功”，不一定表示设备真的完成了下载、烧录和重启，也可能只是因为平台判断“当前版本已经满足目标版本”。

### 2. 设备是否在线

- 设备在线时，平台会立即发起 OTA 协商。
- 设备离线时，平台会等待设备重新上线，再继续 OTA 流程。

### 3. 下载 Token 有时效

根据官方文档，下载包 URL 对应的 Token 有有效期，设备需要在有效期内完成下载和升级。

## 设备端需要完成的动作

设备端不能只实现“版本上报”，完整 OTA 至少要完成以下动作。

### 1. 接收版本查询命令

设备在线后，需要正确接收平台下发的 OTA 版本查询请求。

设备收到后，需要返回当前运行的软件版本/固件版本。

### 2. 接收升级通知

当平台确认设备需要升级后，会继续下发升级包信息。设备端需要能够解析出：

- 目标版本号
- 下载 URL
- Token
- 升级包相关元数据

### 3. 下载升级包

设备端需要通过 HTTPS 下载升级包。

实现时通常需要考虑：

- 网络异常重试
- 下载超时处理
- 断点续传能力
- 存储空间是否足够

### 4. 校验升级包

下载完成后，设备端应校验升级包完整性和合法性，例如：

- 文件大小是否正确
- 摘要/签名是否匹配
- 固件是否适用于当前设备型号

### 5. 执行本地 OTA

对于 ESP32/ESP-IDF，一般需要完成：

- 打开 OTA 写入分区
- 将新固件写入 OTA 分区
- 完成写入并校验
- 设置新的启动分区
- 重启设备

### 6. 重启后上报结果

设备启动到新固件后，需要再次向平台上报：

- 当前实际运行的新版本
- 升级结果（成功/失败）

只有这样，平台才能形成完整的 OTA 闭环。

## 为什么“只上报版本”还不够

只上报版本，最多只能说明设备完成了 OTA 流程中的“版本查询响应”阶段。

它 **不能** 代表以下事情已经发生：

- 升级包已经下发
- 升级包已经下载
- 固件已经写入 Flash
- 设备已经切换到新分区启动
- 设备已经完成升级结果反馈

也就是说：

> “上报版本”只是 OTA 的前半段握手，不是 OTA 升级本身。

## 常见现象说明

### 现象一：平台启动升级任务后，只收到版本查询

这通常说明 OTA 流程还停留在协商阶段。

可能原因：

- 平台先在确认设备当前版本
- 设备当前版本与目标版本一致，因此流程提前结束
- 设备端没有正确处理后续升级消息

### 现象二：平台任务直接显示成功，但设备没有升级

这通常有两种情况：

1. **当前版本和目标版本相同**，平台认为无需升级。
2. **设备端只实现了版本上报，没有实现真正 OTA 执行逻辑**。

### 现象三：收到了升级消息，但设备没有后续动作

这说明设备端缺少以下能力之一：

- 解析升级包信息
- 使用 URL + Token 下载固件
- 调用 ESP-IDF OTA API 写入固件
- 重启后上报升级结果

## 在 ESP32 项目中的落地建议

如果要在本项目中实现完整的华为云 OTA，建议按下面的顺序落地：

### 第一阶段：补齐 OTA 消息处理

- 正确区分版本查询消息和固件升级消息
- 为固件升级消息增加 JSON 解析逻辑
- 提取目标版本、下载地址、Token 等字段

### 第二阶段：补齐下载能力

- 使用 HTTPS 客户端下载升级包
- 支持下载失败重试
- 增加下载过程日志

### 第三阶段：补齐 ESP-IDF OTA 写入流程

- `esp_ota_begin()`
- `esp_ota_write()`
- `esp_ota_end()`
- `esp_ota_set_boot_partition()`
- `esp_restart()`

### 第四阶段：补齐升级完成后的状态回报

- 新固件启动后读取当前版本
- 向平台上报升级结果
- 向平台上报新版本

## 针对本项目的提醒

如果你的设备当前行为是：

1. 收到平台版本查询
2. 上报当前版本
3. 平台任务显示成功

那么要优先检查两件事：

### 1. 目标版本是否与当前版本一致

如果一致，平台直接成功是符合官方流程的。

### 2. 项目是否真的实现了本地 OTA 执行逻辑

如果项目中只有“版本上报”，而没有：

- 升级消息解析
- HTTPS 下载
- `esp_ota_*` 写入
- 重启切换分区
- 升级结果回报

那就说明当前项目还没有实现完整 OTA。

## 调试建议

建议按下面顺序排查：

1. 查看平台升级任务目标版本号。
2. 查看设备当前上报版本号。
3. 确认两者是否一致。
4. 抓取设备收到的 OTA 下行消息内容。
5. 检查是否收到了升级包 URL 和 Token。
6. 检查设备端是否已经实现下载和烧录逻辑。
7. 检查升级完成后是否重新上报了新版本。

## 总结

华为云 MQTT OTA 的本质是一个完整闭环：

**平台发起任务 → 查询版本 → 判断是否升级 → 下发下载信息 → 设备下载并升级 → 设备反馈结果 → 平台确认完成**

因此，设备端如果只做了“版本上报”，那还只是完成了 OTA 的前置协商，不能算真正完成 OTA 升级。
